# ClearVid 视频清晰度增强工具制作文档

## 1. 目标定义

本项目目标是制作一个基于 Python 的本地视频清晰度增强工具，面向真实拍摄、包含人物角色的视频素材，对低分辨率、压缩损伤、噪点、轻微模糊等问题进行综合修复，并尽可能充分利用 RTX 5090 显卡能力。

工具的首期目标不是做学术意义上的单一超分模型复现，而是做一个可稳定落地的工程化处理链路，达到以下结果：

- 输入低分辨率真实视频，例如 360p、480p、540p、720p
- 主要输入以 480p、720p 为主，兼容更低分辨率素材
- 主要输出目标为 1080p 和 4K
- 输出更清晰、更稳定、人物脸部更自然的视频
- 尽量减少闪烁、边缘振铃、过度锐化、皮肤蜡像感
- 在 Windows 本地环境下优先支持单机高性能处理
- 首期支持批量处理与简单界面
- 预留后续 TensorRT 加速、模型切换、跨硬件配置适配能力

## 1.1 当前已确认的项目决策

根据当前确认结果，本项目按以下约束推进：

- 输出目标以 1080p 和 4K 为主
- 输入源以 720p、480p 真实拍摄视频为主
- 项目优先级为画质优先，其次再做性能压榨
- 首版允许提供简单界面，不只做命令行
- 首发环境优先适配 Windows 本机高性能配置
- 重点优化目标机器为 RTX 5090 + 高性能桌面 CPU + 48GB 内存
- 同时保留对其他 Windows 显卡配置的降级运行能力
- 输出保留音频、字幕、封面和可保留的元数据
- 支持单文件与目录批量处理
- 首版不做插帧，只做清晰度增强与稳定性控制
- V1 主方案采用 Real-ESRGAN + CodeFormer 路线

## 2. 需求边界

### 2.1 处理对象

优先处理以下类型视频：

- 真人拍摄视频
- 包含人物半身、全身、面部镜头
- 常见来源包括手机视频、社交平台压缩视频、旧视频转存

### 2.2 首期不优先处理

以下内容不作为首期强保证目标：

- 动画、二次元、卡通视频
- 严重运动模糊无法恢复的素材
- 极端压缩损坏、严重马赛克、丢帧严重的视频
- 电影级调色、影视级修复工作流替代品

## 3. 总体技术路线

建议采用“多阶段增强流水线”而不是单模型一把梭：

1. 视频解码与分析
2. 预处理
3. 基础画质修复
4. 视频超分辨率
5. 人脸区域增强
6. 时序一致性控制
7. 可选插帧
8. 视频编码输出

原因如下：

- 真实视频退化形式复杂，单一模型通常只能解决部分问题
- 人物视频对脸部质量极其敏感，需要单独处理 ROI
- 视频任务不仅是单帧清晰，还需要尽量抑制闪烁和细节抖动
- 工程上可以先做稳定版本，再针对各阶段替换更强模型

## 4. 模型与工具选型建议

## 4.1 基础组件

建议优先采用以下开源免费组件：

- FFmpeg：解码、滤镜、编码、音频保留、封装
- PyTorch：模型推理主框架
- OpenCV 或 PyAV：逐帧读写与中间处理
- facexlib：人脸检测、人脸对齐、面部相关预处理
- ONNX Runtime / TensorRT：后续高性能部署优化

## 4.2 图像级超分基础模型

首选模型：Real-ESRGAN

建议原因：

- 开源成熟，社区广泛使用
- 对真实世界退化比传统 ESRGAN 更稳妥
- 工程集成成本较低，适合作为第一阶段基线
- 可以作为单帧兜底增强模块

建议使用模型：

- RealESRGAN_x4plus：通用真实图像增强主力模型

不建议作为主模型的方向：

- 动漫专用模型，不适合真人视频
- 只针对静态图片、没有视频一致性考虑的轻量模型作为唯一主干

## 4.3 视频超分主模型

建议分两阶段：

### 第一阶段主模型

- Real-ESRGAN 单帧超分 + 时序后处理

优点：

- 开发快
- 依赖成熟
- 容易跑满 GPU
- 便于先做出稳定可用版本

缺点：

- 时序一致性不如原生视频超分模型
- 快速运动和纹理区域可能出现轻微闪烁

### 第二阶段主模型

- RealBasicVSR 或 BasicVSR++ 路线
- 可选研究 RVRT 路线作为更高质量候选

建议判断：

- 如果首要目标是尽快落地，先做 Real-ESRGAN 工程版
- 如果首要目标是追求视频一致性和更高上限，再引入 BasicVSR++ 类模型

推荐决策：

- V1：Real-ESRGAN 为主
- V2：增加 RealBasicVSR / BasicVSR++ 模式
- V3：尝试 TensorRT 化和混合调度

## 4.4 人脸修复模型

建议主选：CodeFormer

备选：GFPGAN

推荐原因：

- 人物视频的主观质量很大程度取决于脸部
- CodeFormer 对真实人物脸部的细节重建与自然度平衡通常更好
- GFPGAN 集成成熟，适合做可切换备选模式

建议策略：

- 默认仅对检测到的人脸区域做修复
- 提供保守、标准、增强三个强度档位
- 人脸修复后进行融合，避免整帧统一修脸导致风格漂移

## 4.5 去噪 / 去块效应 / 预处理

首期建议优先采用工程化组合，而不是再引入过多大模型：

- FFmpeg 预处理滤镜
- 轻量级锐化与降噪参数控制
- 可选去隔行检测与处理

建议首期预处理项：

- 去隔行：适用于老旧视频
- 轻度降噪：降低压缩噪点对超分模型的误导
- 色彩空间标准化：统一模型输入
- 场景切换识别：避免跨镜头时序串扰

## 4.6 插帧策略

当前项目阶段不纳入插帧。

原因：

- 你的当前目标是清晰度提升，不是运动流畅度提升
- 插帧会显著增加工程复杂度、显存占用和调参成本
- 对真实人物视频而言，插帧错误比轻微不插帧更容易被察觉

结论：

- V1 和 V2 不做插帧功能
- 后续如有需要，再将 RIFE 作为独立可选模块接入

## 5. 推荐的首版处理链路

建议首版流水线如下：

1. FFmpeg 探测输入信息
2. NVDEC 解码到帧序列
3. 进行场景切分或关键帧边界识别
4. 轻度预处理
5. 使用 Real-ESRGAN 做基础超分
6. 检测人脸并用 CodeFormer 做脸部修复
7. 将脸部结果按掩码融合回整帧
8. 做轻量时序稳定处理
9. 使用 NVENC 编码输出并保留原音频、字幕与可继承元数据

这条链路的优点：

- 能较快产出第一个可用版本
- 能解决人物视频最关键的脸部质量问题
- 能为后续替换更高级的视频超分模型保留接口

## 6. 为什么不是一开始就只上 BasicVSR++

虽然 BasicVSR++ 等模型在视频一致性上更有优势，但不建议作为第一步唯一方案，原因如下：

- 训练和推理依赖链更复杂
- 真正面向真实压缩视频时，前后处理要求更高
- 直接工程化成本明显高于 Real-ESRGAN
- 首期更需要稳定、可维护、容易调参的结果

更稳妥的策略是：

- 先做一个高质量单帧增强 + 脸部修复 + 时序控制版本
- 有 baseline 结果后，再并入视频主干模型做 A/B 测试

## 7. 5090 显卡利用方案

本项目应明确按“先 PyTorch 高质量跑通，再 TensorRT 压榨”的路线设计。

## 7.1 首期 GPU 利用策略

- 使用最新稳定 PyTorch + CUDA 12.x
- 推理优先使用 FP16 或 BF16
- 默认优先保证质量，不为追求极限吞吐而牺牲明显观感
- 批量帧处理，避免单帧频繁上下文切换
- 数据读取、预处理、推理、编码做异步流水线
- 尽量避免 CPU 与 GPU 之间频繁来回拷贝
- 解码和编码优先走 NVDEC / NVENC

## 7.2 第二阶段优化策略

- 将 Real-ESRGAN、CodeFormer、RIFE 导出 ONNX
- 将 Real-ESRGAN、CodeFormer 导出 ONNX
- 对稳定模型建立 TensorRT Engine
- 使用固定输入尺寸或分桶尺寸减少重建开销
- 使用 CUDA Stream 并行不同阶段任务
- 对 ROI 脸部修复采用小批量并发推理

## 7.4 面向不同 Windows 配置的适配策略

虽然首要目标机器是 RTX 5090，但架构上需要提前考虑其他 Windows 用户：

- 高端档：RTX 5090 / 5080，开启大 tile、高 batch、FP16、并行流水
- 中端档：4070 / 4080 / 5070 级别，降低 batch 与 tile，保留主链路
- 入门档：显存较小的 NVIDIA 显卡，自动降级为更小 tile 和单帧批次
- 非 TensorRT 环境：保持 PyTorch 路线可运行

工程原则：

- 所有高性能优化都必须是可选增强，而不是运行前提
- 质量模式与硬件模式分离，避免硬件差异直接影响用户理解

## 7.3 5090 重点压榨方向

真正能把 5090 吃满的不是单一模型，而是完整流水线设计：

- 大分辨率 tile 推理和多 tile 并发
- 批量帧推理
- 解码、推理、编码并行
- 多模型流水线并行
- 视频分段后多进程调度

如果后续样片以 1080p 输出为主，5090 的优势会很明显；如果只是小分辨率短视频，瓶颈未必在纯推理，反而可能在解码、编码和数据搬运。

## 9. 核心模块设计

## 9.1 输入分析模块

职责：

- 读取分辨率、帧率、编码格式、码率、时长
- 判断是否需要去隔行
- 判断是否低帧率
- 判断是否需要裁切黑边

输出：

- 统一的 VideoMetadata 对象
- 给后续流水线提供处理建议

## 9.2 解码模块

职责：

- 优先用 FFmpeg + NVDEC 解码
- 支持逐帧或分段读取
- 支持保留时间戳

关键要求：

- 稳定性优先于花哨封装
- 首期允许 FFmpeg 子进程方案，后续再做更深集成

## 9.3 超分模块

职责：

- 执行 Real-ESRGAN 或其他模型的主超分
- 支持 tile 推理
- 支持批量推理
- 支持不同缩放倍数与输出策略

建议输出模式：

- scale=2
- scale=4
- target_height / target_width 优先模式

针对你的目标场景，建议默认优先支持：

- 480p -> 1080p
- 720p -> 1080p
- 720p -> 4K

其中 720p -> 4K 不建议简单固定 4x 后直接输出，后续应评估是否需要分阶段增强或更保守锐化策略，避免假细节过重。

## 9.4 人脸增强模块

职责：

- 检测人脸框
- 对人脸区域对齐
- 用 CodeFormer 做局部恢复
- 融合回原帧

关键点：

- 需要限制最大人脸增强强度，避免塑料感
- 多脸场景需要稳定排序与追踪
- 镜头切换时要及时重置缓存

## 9.5 时序稳定模块

首期不建议做过重的研究型模块，可以先采用轻量策略：

- 基于光流或特征的相邻帧一致性约束
- 对人脸修复结果做保守融合
- 场景切换时重置时序状态

目标不是完全消灭闪烁，而是在不引入明显拖影的前提下降低抖动。

## 9.6 编码输出模块

建议优先支持：

- H.264 NVENC
- HEVC NVENC
- AV1 NVENC

建议特性：

- 保留原音频
- 尽量保留原字幕流
- 尽量保留封面与容器元数据
- 保留原时间基和容器元信息的必要部分
- 提供质量优先、平衡、速度优先三档

## 10. 推荐的 V1 功能范围

V1 只做必要闭环，但会包含一个简单桌面界面，不追求复杂产品化。

建议 V1 功能：

- 命令行工具
- 简单桌面 GUI
- 输入单个视频文件
- 支持目录批量处理
- 输出增强后视频
- 支持 2x 和 4x
- 支持目标分辨率模式
- 重点支持输出 1080p 与 4K
- 支持人物脸部增强开关
- 支持质量模式切换
- 支持 GPU 半精度推理
- 支持 NVDEC / NVENC
- 支持音频、字幕、元数据保留
- 支持基本日志与进度显示

V1 不建议一开始就做：

- Web UI
- 多机分布式
- 自动模型下载中心
- 复杂任务队列系统

## 11. 推荐的质量模式设计

建议暴露三个模式：

### fast

- Real-ESRGAN 单帧增强
- 关闭或弱化脸部修复
- 更激进的批量和 tile 并发

### balanced

- Real-ESRGAN + CodeFormer
- 轻量时序稳定
- 默认推荐模式

### quality

- 更保守的降噪
- 更高质量的人脸融合
- 可选视频超分主干或更慢的后处理

考虑到你明确要求画质优先，建议产品默认模式直接设为 quality，而不是 balanced。

## 13. 推荐的评估与验收标准

除了主观观感，建议建立基础验收指标：

- 是否明显提升人物面部可辨识度
- 是否明显减少压缩糊感
- 是否出现严重闪烁
- 是否出现过度磨皮
- 是否存在明显鬼影、边缘撕裂、字幕异常
- 输出时长与输入时长是否一致
- 音频是否正常保留

建议准备一个样片集：

- 近景人脸
- 中景人物
- 夜景高噪点
- 快速运动
- 社交平台强压缩视频

建议额外补充两类样片：

- 720p 升 4K 的细节极限样片
- 含字幕、封面或多音轨的复杂封装样片

### 里程碑 M4

- 引入 BasicVSR++ 或 RealBasicVSR 模式

### 里程碑 M5

- ONNX / TensorRT 部署优化
- 最大化利用 5090 性能

## 15. 风险与难点

主要风险如下：

- 单帧超分容易带来时序闪烁
- 人脸修复强度过大时容易失真
- 不同来源视频退化分布差异极大
- 模型组合过多会让调参复杂度明显上升
- 5090 的极致性能发挥依赖 CUDA、PyTorch、TensorRT 版本匹配

因此不建议一开始就追求“最强模型全堆上去”，而是先建立一条稳健、可调、可测的主链路。

## 16. 首版推荐技术决策

如果现在立项，我建议直接采用以下方案作为 V1：

- 语言：Python 3.11
- 框架：PyTorch
- 解码编码：FFmpeg + NVDEC/NVENC
- 主超分：RealESRGAN_x4plus
- 人脸修复：CodeFormer
- 可选备选脸修复：GFPGAN
- 命令行：Typer
- 简单桌面界面：PySide6
- 配置：YAML
- 首发平台：Windows 本地

更具体的产品决策如下：

- 默认目标是高质量增强，不把速度作为首要优化目标
- 默认输出目标档位为 1080p 与 4K
- 默认保留音频、字幕和可继承元数据
- 默认支持目录批处理
- 默认不实现插帧
